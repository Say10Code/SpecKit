# Transmission Protocols — T=0 и T=1

## Определение

**T=0** и **T=1** — два транспортных протокола, определённых в ISO/IEC 7816-3 и адаптированных ETSI TS 102 221, для передачи APDU между терминалом и UICC. ^[extracted]

## T=0 — Байт-ориентированный полудуплекс

### Характеристики

- **Полудуплекс**: в каждый момент передаёт либо терминал, либо UICC
- **Байт-ориентированный**: обработка на уровне байт
- **Procedure bytes**: UICC управляет потоком через байты процедуры в ответах
- Используется в **GSM SIM** (legacy), поддерживается всеми UICC

### Procedure Bytes

В ответ на каждый байт команды UICC может вернуть:

| Byte | Значение | Действие терминала |
|---|---|---|
| `INS` | Эхо инструкции (ACK) | Продолжить отправку данных |
| `INS^0xFF` | Complement ACK | Продолжить отправку данных |
| `60` | NULL | Ожидание следующего procedure byte |
| `61 XX` | SW1, XX байт доступно | GET RESPONSE с Le=XX |
| `6C XX` | Wrong Le, XX = правильный | Повторить команду с Le=XX |
| `90 00` | Нормальное завершение | Конец команды |

### Особенности T=0

1. **Case 2/4 требует GET RESPONSE**: Если команда ожидает данные в ответе (Case 2/4), UICC возвращает `61 XX` → терминал выполняет GET RESPONSE (INS=`C0`) для получения данных ^[extracted]
2. **Нет встроенной обработки ошибок передачи**: Ошибки детектируются по неправильному parity на I/O
3. **Нет мультиплексирования каналов**: Логические каналы эмулируются через CLA

### Пример T=0 (Case 2: READ BINARY)

```
T→UICC: CLA INS P1 P2 Le     (команда)
UICC→T: INS                   (ACK — procedure byte)
UICC→T: 61 XX                 (XX bytes available)
T→UICC: 00 C0 00 00 XX        (GET RESPONSE)
UICC→T: Data ...Data SW1 SW2  (данные + статус)
```

## T=1 — Блочно-ориентированный полнодуплекс

### Характеристики

- **Полнодуплексный**: терминал и UICC могут обмениваться блоками в обе стороны
- **Блочная структура**: данные передаются целыми блоками с проверкой ошибок
- **EDC (Error Detection Code)**: LRC или CRC для каждого блока
- **Chaining**: цепочки блоков для больших данных
- **Надёжность**: retransmission при ошибках

### Структура блока

```
┌──────────┬─────────────┬──────────┬──────────┐
│ Prologue │ Information │ Epilogue │          │
│ (3 bytes)│ (IFS bytes) │ (1 byte) │          │
└──────────┴─────────────┴──────────┴──────────┘
│ NAD PCB LEN │   Данные    │ EDC      │
```

- **NAD** (Node Address): Адрес источника/назначения
- **PCB** (Protocol Control Byte): Тип блока + управление
- **LEN**: Длина Information field
- **EDC**: Error Detection Code (LRC — 1 byte, или CRC — 2 bytes)

### Типы блоков

| Тип | PCB pattern | Назначение |
|---|---|---|
| **I-block** | ...0..... | Данные (информационный блок) |
| **R-block** | ...1..0.. | Квитанция (ACK/NACK) |
| **S-block** | ...1..1.. | Управление (RESYNC, IFS, ABORT, WTX) |

### Тайминги T=1

| Параметр | Значение | Описание |
|---|---|---|
| **CWT** (Character Waiting Time) | (CWI+1) × etu | Максимальное время между символами в блоке |
| **BWT** (Block Waiting Time) | (BWI+1) × etu + 12 | Максимальное время между блоками |
| **BGT** (Block Guard Time) | Минимальное время между блоками | |

### Chaining (цепочки блоков)

Если данные не помещаются в один I-block (больше IFS), используется chaining:
- Bit 5 (M-bit) в PCB: `1` = more blocks follow, `0` = last block
- Вся цепочка собирается перед передачей на транспортный уровень

### Преимущества T=1 над T=0

| T=0 | T=1 |
|---|---|
| Полудуплекс | Полный дуплекс |
| Байт-ориентированный | Блочный |
| GET RESPONSE для Case 2/4 | Прямая передача данных в I-block |
| Простая обработка ошибок | Retransmission при ошибках (EDC) |
| Меньше накладных расходов | Больше overhead (NAD, PCB, LEN, EDC) |
| Проще в реализации | Сложнее, но надёжнее |

## Транспортный уровень: транспорт APDU

Над T=0 и T=1 работает транспортный уровень, который отображает APDU на TPDU (Transfer Protocol Data Unit). Детали маппинга специфичны для каждого протокола (cases 1-4, chaining, procedure bytes).
