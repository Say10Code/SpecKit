# eSIM: От UICC к встроенному профилю

> **Synthesis** — эволюция от физической SIM-карты к удалённому управлению профилями. Как работает, почему два решения, что изменилось.

---

## 1. Три эпохи SIM

```
Эпоха I: Физическая SIM     Эпоха II: UICC          Эпоха III: eUICC (eSIM)
═══════════════════════     ════════════════════     ═══════════════════════
1991-2000                   2000-2016                2016-настоящее

  ┌─────┐                     ┌─────┐                  ┌──────────────┐
  │ SIM │  1 профиль          │UICC │  N приложений    │ eUICC       │
  │     │  = 1 оператор        │     │  = N ADF         │ ┌────┬────┐ │
  └─────┘                     └─────┘                  │ │Prof│Prof│ │
      ↓                            ↓                   │ │ A  │ B  │ │
  Смена =                        Смена =              │ └────┴────┘ │
  новая карта                    новая карта          └──────────────┘
                                                           ↓
                                                      Смена =
                                                      загрузка профиля OTA
```

---

## 2. Что изменилось

| Свойство | UICC (классическая) | eUICC (eSIM) |
|---|---|---|
| **Носитель** | Пластик (ID-1/2FF/3FF/4FF) | Встроенный чип (MFF2) или съёмный |
| **Профили** | 1 профиль на карте | Множественные профили |
| **Смена оператора** | Физическая замена | OTA-загрузка профиля |
| **Производство** | Изготовление + персонализация | Чип + удалённая загрузка |
| **Логистика** | Физическая дистрибуция | Цифровая (QR-код) |
| **Безопасность** | Ключи оператора в чипе | PKI + сертификаты + SAS |
| **Стандарты** | ETSI TS 102 221 | GSMA SGP.02 / SGP.22 |

## 4. Два решения: M2M vs Consumer

### M2M (SGP.02) — Push-модель

```
Оператор → SM-DP → SM-SR → eUICC
           (подготовка)  (маршрутизация)  (установка)

Без участия пользователя. Сервер решает, какой профиль
установить, активировать, удалить.
```

**Где используется**: счётчики энергии, automotive, трекеры, промышленный IoT.

| Элемент | Роль |
|---|---|
| **SM-DP** | Создаёт профиль, шифрует, подписывает |
| **SM-SR** | Хранит ключи eUICC, доставляет профиль, управляет статусом |
| **eUICC** | Чип с ISD-P для каждого профиля |
| **Аутентификация** | PSK (Pre-Shared Key) |
| **Транспорт** | SMS, CAT_TP, TCP/IP (BIP) |
| **SM-SR Swap** | Да — передача eUICC другому SM-SR |

### Consumer (SGP.22) — Pull-модель

```
Пользователь → сканирует QR → LPA → SM-DP+ → eUICC
                                     (загрузка)

Пользователь управляет профилями через UI устройства.
```

**Где используется**: смартфоны (iPhone XS+, Pixel 4+, Galaxy S20+), планшеты, часы.

| Элемент | Роль |
|---|---|
| **SM-DP+** | Создаёт профиль + доставляет (+ функции SM-SR) |
| **SM-DS** | "Доска объявлений" для уведомлений |
| **LPA** | UI на устройстве: загрузка + управление профилями |
| **eUICC** | Чип с поддержкой потребительских профилей |
| **Аутентификация** | PKI (X.509 сертификаты) |
| **Транспорт** | HTTPS/IP |
| **Companion devices** | Часы через телефон |

## 5. Профиль (Profile) — душа eSIM

```
Profile Package (SIMalliance Interoperable):
├── ISD-P (Issuer Security Domain - Profile)
│   └── Ключи и сертификаты профиля
├── File System
│   ├── MF (3F00)
│   │   ├── EF_ICCID (2FE2)
│   │   ├── EF_DIR (2F00)
│   │   └── EF_PL (2F05)
│   └── ADF.USIM
│       ├── EF_IMSI (6F07)
│       ├── EF_Keys (6F08)
│       ├── EF_SPN (6F46)
│       └── ... (все EF из TS 31.102)
├── Security Keys (Ki, K, OPc)
├── Applications (USIM, ISIM, STK-апплеты)
└── Profile Metadata
    ├── Profile Name
    ├── Operator Name (SPN)
    ├── Profile Class
    └── Policy Rules (когда можно disable/delete)
```

> 🔑 **Ключевой факт**: Профиль содержит ТУ ЖЕ файловую систему, что и физическая SIM-карта. Разница лишь в способе доставки.

## 6. Жизненный цикл профиля

```
    ┌──────────┐
    │ Created  │  SM-DP(+) создал профиль, но ещё не на eUICC
    └────┬─────┘
         │ Download & Install
         ▼
    ┌──────────┐
    │ Disabled │  Профиль загружен, но не активен
    └────┬─────┘
         │ Enable (пользователь / сервер)
         ▼
    ┌──────────┐
    │ Enabled  │  Активный профиль — устройство в сети оператора
    └────┬─────┘
         │ Disable / Delete
         ▼
    ┌──────────┐
    │ Deleted  │  Профиль удалён с eUICC
    └──────────┘
```

- Только **один профиль Enabled** одновременно (как одна SIM-карта)
- Можно Disable профиль оператора А и Enable профиль оператора Б
- Профиль остаётся собственностью оператора (лицензия)

## 7. Безопасность: три уровня

### Уровень 1: Чип (eUICC)
- Common Criteria **EAL4+** сертификация
- Protection Profile: eUICC устойчив к физическим атакам
- ISD-P изолирует профили друг от друга

### Уровень 2: Производство (SAS)
- **SAS-UP**: сертификация площадок персонализации eUICC
- **SAS-SM**: сертификация SM-DP/SM-DP+ хостинга
- Без SAS — нельзя участвовать в экосистеме GSMA

### Уровень 3: Коммуникация (PKI)
- **GSMA CI** (Certificate Issuer) — корневой центр сертификации
- Каждый SM-DP+ и eUICC получают сертификаты
- Взаимная аутентификация при загрузке профиля
- Consumer: любой eUICC ↔ любой SM-DP+ (общий root CA)
- M2M: PSK — только свой SM-SR

## 8. Сравнение: UICC vs eUICC

| Операция | UICC | eUICC |
|---|---|---|
| **Получить связь** | Купить SIM → вставить | QR-код → загрузка профиля |
| **Сменить оператора** | Купить новую SIM → вставить | Новый профиль OTA |
| **Два номера** | Два слота SIM | Множественные профили |
| **Путешествия** | Роуминг или местная SIM | Местный профиль OTA |
| **Потеря устройства** | Блокировка SIM | Удалённое удаление профиля |
| **IoT (1000 устройств)** | 1000 SIM-карт | 1000 eUICC + OTA |

---

## 9. Что eSIM меняет для разработчика

| Аспект | Раньше (UICC) | Теперь (eUICC) |
|---|---|---|
| **Файловая система** | Та же | Та же! (TS 102 221) |
| **APDU** | Те же | Те же |
| **STK/JavaCard** | Те же API | Те же API |
| **Установка апплетов** | GP через PCSC | GP + OTA (SMS/BIP/HTTPS) |
| **Тестирование** | Физическая карта | eUICC + Test Profile |
| **Инструменты** | pySim, gp.jar | pySim, gp.jar (те же!) |

> 💡 **Для разработчика STK-апплетов**: файловая система, APDU-команды и JavaCard API НЕ изменились. eSIM добавляет уровень управления профилями — но не меняет логику внутри профиля.

---

## 10. Будущее: SGP.32 (IoT)

GSMA разрабатывает **SGP.32** — eSIM для IoT, объединяющее лучшие стороны M2M и Consumer:
- Не требует LPA (как M2M)
- Не требует SM-SR (как Consumer)
- Поддержка множественных операторов без SM-SR Swap
- Легковесный протокол для constrained устройств
