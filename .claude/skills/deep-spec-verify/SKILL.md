# Skill: deep-spec-verify
# Trigger: /deep-spec-verify

Универсальная перекрёстная верификация ЛЮБОЙ технической статьи с эталонными текстами спецификаций. **Гибрид AI + скрипт**: скрипт делает механическую работу (извлечение, grep, сверка), AI делает семантическое суждение (вымысел? логическая связность?).

Работает для статей любого домена — UICC, RRC, NAS, JavaCard, STK, Security, OTA, etc.

## Синтаксис

```
/deep-spec-verify <путь-к-статье> [--fix]
```

**Примеры**:
- `/deep-spec-verify wiki/syntheses/ue_capability_information.md`
- `/deep-spec-verify wiki/syntheses/sim_files_5g.md --fix`

## Workflow

### Phase 1: Скрипт — механическая верификация

```bash
python "D:\ObsidianDB\_tech\scripts\verify_spec_claims.py" "<путь-к-статье>" [--fix]
```

Скрипт **доменно-независим**:
1. **Извлекает** все утверждения по 9 структурным паттернам: hex-значения, секционные ссылки, кодовые блоки, строки таблиц, key=value, числовые значения, кодирования, структурные связи, prose-кандидаты
2. **Строит SpecIndex** — индекс всех spec-файлов (hex→spec, keyword→spec, section→spec). Кэшируется в `.speckit/spec_index.json`
3. **Верифицирует** каждое утверждение: grep по спек-файлам, сверка значений
4. **Детектирует аномалии** — чисто статистические сигналы возможного вымысла
5. **Сохраняет отчёт** в `<статья>.verify.md`

### Phase 2: AI — семантическое суждение

Прочитай отчёт `<статья>.verify.md`. Для каждого Finding с вердиктом:
- `FABRICATION_FLAG` — скрипт заподозрил вымысел
- `MISMATCH` — значение не совпадает со спецификацией
- `NOT_FOUND` — значение не найдено в спеке
- `AMBIGUOUS` — prose-утверждение, требует AI-проверки

Следуй инструкциям из [ai_judge.md](ai_judge.md):

1. **Прочитай релевантный участок спека** из `specs-extracted/` (используй `spec_file` и `spec_line` из Finding)
2. **Сравни смысл утверждения** с текстом спецификации
3. **Вынеси AI-вердикт**: `CONFIRMED_FABRICATION` / `CONFIRMED_MISMATCH` / `LEGITIMATE` / `SPEC_AMBIGUOUS` / `NEEDS_DEEPER_CHECK`
4. **Сгенерируй конкретное исправление** для подтверждённых ошибок
5. **Проверь логическую связность** — внутренние противоречия в статье

### Phase 3: Исправление

Примени исправления через `Edit`:
- **Детерминированные** (hex-опечатки, номера секций) — применяй сразу
- **AI-подтверждённые** (CONFIRMED_MISMATCH, CONFIRMED_FABRICATION) — применяй с указанием обоснования
- **Спорные** (SPEC_AMBIGUOUS, NEEDS_DEEPER_CHECK) — отметь в статье `> [!note] NEEDS REVIEW: ...` для ручной проверки

### Phase 4: `/lint`

После всех исправлений — проверь wikilinks и frontmatter.

## Совместное использование

```
/deep-spec-verify article.md     ← Этот скилл (факты + вымысел)
/review article.md               ← Reviewer v3 (структура + связность)
/lint                            ← Wikilinks + frontmatter
```

## Что проверяется

| Тип утверждения | Пример | Метод |
|---|---|---|
| **hex_value** | `FID = 0x6F07` | Grep в спеке → точное совпадение |
| **section_ref** | `^[TS 31.102 §4.2.1]` | Проверка существования секции |
| **code_block** | `` ```asn1 UE-NR-Capability ::= ... ``` `` | Проверка существования типа в спеке |
| **table_row** | `\| 6F07 \| EF_IMSI \| 9 bytes \|` | Сверка значений ячеек со спеком |
| **key_value** | `SRB = SRB1`, `Direction = UE → Network` | Поиск пары ключ-значение в спеке |
| **numeric** | `9 bytes`, `size = 28` | Поиск числового утверждения |
| **encoding** | `PER-кодирование`, `CSN.1` | Проверка формата кодирования |
| **structural** | `EF_IMSI тип = Transparent` | Проверка классификации |
| **prose** | `Согласно TS 31.102, EF_IMSI хранит IMSI` | AI-проверка семантики |

## Интерпретация вердиктов

| Вердикт скрипта | Значение | Действие |
|---|---|---|
| `VERIFIED` | Подтверждено спецификацией | Ничего |
| `MISMATCH` | Противоречит спецификации | AI-подтверждение → исправить |
| `FABRICATION_FLAG` | Статистический сигнал вымысла | AI-подтверждение → исправить |
| `NOT_FOUND` | Не найдено в спеке | Проверить другой спек / версию |
| `NEEDS_SPEC` | Спецификация недоступна | `/spec-download` → SpecExtractor |
| `AMBIGUOUS` | Prose, требует AI | AI читает спек → вердикт |

## Требования к спецификациям

Скрипт автоматически находит спецификации в:
- `specs-extracted/` — все TXT/MD/JSON
- `Specifications/!INCOMING/` — свежескачанные

Если спецификация не найдена — скачай через `/spec-download` и извлеки через SpecExtractor.
