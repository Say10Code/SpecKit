### 3.1 «Золотой путь» — /spec-download

```

  │
  ├─[1] SpecDownloader

  │     └── python -m _pipeline download 31.102            (3GPP FTP → !INCOMING/)
  │     ⏱️ ~10 сек (metadata) + ~5 сек (download .docx)
  │
  ├─[2] Librarian
  │     ├── Сканирование !INCOMING/Specs/archive/
  │     ├── Flatten: .docx → Specifications/<тема>/        (по .category-map.md)
  │     └── Удаление Specs/ из !INCOMING/
  │     ⏱️ ~5 сек
  │
  ├─[3] Author v2 Batch (1 вызов на спецификацию)
  │     ├── Чтение PDF/извлечённого текста (1 раз)
  │     ├── Создание pages: summary → concepts → entities
  │     ├── Обновление индексов (wiki/index.md + раздельные)

  │
  ├─[4] SpecExtractor / extract_docx.py
  │     ├── .docx → extract_docx.py --tables               (Tier 1, 0.2 сек)

  ├─[5] Linker (1 проход после ВСЕХ Author)
  │     ├── Поиск orphan
```

## 4. Узкие места

### 🔴 BOTTLENECK 1: Author — единственный writer в wiki/

**Ситуация**: 6 из 8 рабочих процессов проходят через Author. Никто другой не создаёт wiki-страницы.
- Librarian → Author
- /ingest → Author
- /spec-download → Librarian → Author
- Researcher → Author (Single mode)
- /research → Researcher → Author

**Последствие**: Author — single point of failure. Если Author занят (обрабатывает большой PDF), новый /spec-download будет ждать.

**Степень**: 🟡 Средняя. Batch mode решил проблему N последовательных вызовов → 1 вызов. Но всё ещё один агент на всё.

### 🟡 BOTTLENECK 2: /lint — универсальный пост-процессор

**Ситуация**: Каждый агент после записи в wiki/ вызывает /lint. Это 5+ вызовов /lint на один /spec-download.

**Последствие**: /lint запускается избыточно. После Author (создал страницы) → /lint. После Linker (добавил ссылки) → /lint. Это два вызова подряд, второй инвалидирует первый.

**Предложение**: Объединить /lint в конце цепочки, а не после каждого шага.

### 🟡 BOTTLENECK 3: Reviewer — изолирован от пайплайна

**Ситуация**: Reviewer НЕ вызывается автоматически после /spec-download. Он вызывается только вручную через /review.

**Последствие**: Новые страницы создаются, но не рецензируются. Quality Score падает со временем. Reviewed % = 94.6% (не 100%).

**Это НЕ баг, а фича**: Reviewer требует человеческого суждения (CRITICAL/HIGH/MEDIUM/LOW). Авто-вызов без человека опасен — ложные CRITICAL.

### 🔵 BOTTLENECK 4: SpecExtractor pre-flight

**Ситуация**: `auto_patch_docling.py` — обязательный pre-flight перед Docling. Если патч не применился → Docling крашится с bad_alloc.

**Последствие**: Хрупкая зависимость от внутренней структуры docling. При обновлении docling патч может перестать работать.

**Митигация**: F1 fix уже заапстримлен? Проверить. Если нет — следить за версиями docling.

## 5. Точки отказа (Single Points of Failure)

| Компонент | Критичность | Что ломается | Восстановление |
|---|---|---|---|
| **Author v2** | 🔴 Критический | Ни одна wiki-страница не создаётся | Перезапуск. Batch mode: repeatable. |
| **_pipeline/** | 🟡 Высокая | Нельзя скачать новые спецификации | Ручное скачивание с 3GPP FTP |
| **.speckit/ БД** | 🟢 Средняя | metadata fetch не работает → download без --release может выбрать не ту версию | Пересоздать: `python -m _pipeline metadata fetch` |
| **auto_patch_docling** | 🟢 Средняя | Docling падает с bad_alloc. PyPDF2 fallback работает | Перезапустить auto_patch |
| **Reviewer** | 🟢 Низкая | Новые страницы без ревью. Данные не теряются | Запустить /review когда Reviewer доступен |
| **Linker** | 🟢 Низкая | Новые страницы — сироты. Исправляется следующим запуском | Запустить Linker отдельно |

## 10. Рекомендации

### R1: Объединить /lint в конце цепочки

Сейчас /lint вызывается после каждого шага. Предложение: один вызов в конце цепочки (после Linker). Экономия: 3-4 лишних вызова /lint на каждый /spec-download.

### R2: Добавить авто-триггер Reviewer для новых страниц

После /spec-download + /lint — предложить пользователю `/review <новая страница>`. Не авто-вызов (опасно), а напоминание.

### R3: Мониторинг auto_patch_docling

Добавить в SpecExtractor pre-flight проверку версии docling. Если версия изменилась — предупредить что патч может не примениться.

### R4: Linker — батч-режим

Сейчас Linker вызывается после каждого Author. Для /spec-download с 3+ спецификациями — вызывать Linker один раз после ВСЕХ Author (уже реализовано в P2-5).
