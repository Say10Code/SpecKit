# Batch Authoring vs Author Split — анализ пользы и рисков

> **Дата**: 2026-06-13 19:00
> **Метод**: Трассировка всех 5 call sites Author, анализ реальных wiki-страниц, оценка контекстного окна
> **Вердикт**: Batch Authoring ✅ внедрять. Author Split ❌ не внедрять.

---

## 1. Что делает Author сейчас

Author — AI-агент (61 строка system prompt), создающий wiki-страницы 4 типов:

| Тип | Шаблон | Объём | Частота |
|---|---|---|---|
| summary | t-summary | 2-5 KB | 1 на спецификацию |
| concept | t-concept | 2-4 KB | 3-5 на спецификацию |
| entity | t-entity | 1-2 KB | 0-1 на спецификацию |
| synthesis | t-synthesis | 5-15 KB | 0-1 на спецификацию |

Реальная производительность (измерено по истории):
- 1 страница: 45-120 сек
- 1 спецификация (4-7 страниц): 3-8 мин последовательно
- 3 спецификации (12-21 страница): 9-25 мин

## 2. Batch Authoring: анализ

### Как работает сейчас (5 вызовов на TS 31.102)

```
1. Agent(Author: summary TS 31.102)  ── читает PDF (200 стр) → пишет 2KB summary
2. Agent(Author: concept UICC)       ── читает PDF (200 стр) → пишет 2KB concept
3. Agent(Author: concept USIM)       ── читает PDF (200 стр) → пишет 2KB concept
4. Agent(Author: concept EF)         ── читает PDF (200 стр) → пишет 3KB concept
5. Agent(Author: concept ADF)        ── читает PDF (200 стр) → пишет 2KB concept
                                      ──
                                      PDF прочитан 5 раз.
                                      Контекст потерян между вызовами.
```

### Как будет работать с Batch

```
1. Agent(Author v2: пакет TS 31.102)
   ├─ Читает PDF ОДИН раз (200 стр → ~250K знаков)
   ├─ Планирует: summary + 4 концепта
   ├─ Пишет summary ── ссылается на концепты (уже знает их имена)
   ├─ Пишет concept/UICC ── ссылается на summary (уже знает структуру)
   ├─ Пишет concept/USIM ── консистентная терминология
   ├─ Пишет concept/EF ── переиспользует определения из summary
   └─ Пишет concept/ADF
```

### Польза ✅

| Фактор | Сейчас | Batch | Выигрыш |
|---|---|---|---|
| Чтений PDF | 5 раз | 1 раз | **5× меньше токенов на чтение** |
| Agent-вызовов | 5 | 1 | **5× меньше overhead** |
| Кросс-ссылки | Ручные (Linker потом) | Автоматически при создании | **Связнее граф** |
| Терминология | Может разойтись | Единая в рамках пакета | **Выше качество** |
| Время на TS 31.102 | ~5 мин | ~1.5 мин | **3.3× быстрее** |
| Транзакционность | Нет (страницы созданы, связи — нет) | Да (все или ничего) | **Чище состояние** |

### Риски ❌

| Риск | Оценка | Mitigation |
|---|---|---|
| **Контекстное окно**: PDF 200+ стр может переполнить | 🟡 Реальный | Гибрид: если PDF > 150K знаков → sequential mode |
| **Размытие качества**: 6 страниц за раз хуже чем 6 по одной | 🟡 Возможен | Явная структура в промите: «сначала summary, потом concepts» |
| **Blast radius ошибки**: одна галлюцинация → все страницы | 🟡 Реальный | Reviewer проверяет все страницы независимо. + provenance-пометки |
| **Сложнее дебажить**: какая страница сломалась? | 🟢 Низкий | Author пишет страницы последовательно, ошибка на N-й не ломает 1..N-1 |
| **Токен-стоимость концентрация**: 1 большой вызов вместо 5 средних | 🟢 Низкий | Токены на чтение PDF экономятся 5×; общая стоимость ~одинакова |

### Вывод по Batch Authoring

**✅ Внедрять.** Решает реальную проблему (N последовательных вызовов). 3-5× быстрее. Контекст переиспользуется. Качество выше из-за консистентности.

**Гибридная стратегия**: если PDF < 150K символов (~100 стр спецификации) → batch mode. Если больше → sequential mode с переиспользованием контекста (сохранить summary и опираться на него для concepts).

## 3. Author Split (Drafter + Editor): анализ

### Что предлагается

```
Author (сейчас)           →     Drafter (новый)    +    Editor (новый)
├─ create summary                ├─ create (быстро,       ├─ update (медленно,
├─ create concepts               │   без проверки          │   с проверкой
├─ create entities               │   дубликатов)           │   дубликатов)
├─ create synthesis              │                         │
├─ update index                  │                         │
└─ check duplicates              │                         │
```

### Реальная картина в ObsidianDB

Сколько страниц в wiki реально обновляются (update) vs создаются (create)?

```
129 wiki-страниц, 100% reviewed
  ├─ Созданы один раз:          ~125 (97%)
  └─ Обновлялись после:          ~4  (3%) — индексы, Roadmap
```

**97% операций Author — это create, не update.** Editor будет простаивать.

### Почему НЕ нужно разделять

| Аргумент | Детали |
|---|---|
| **Проблема не в duplicate check** | Проверка «нет ли уже такой страницы» — это `Glob("wiki/concepts/UICC.md")` — 0.5 сек. Не bottleneck. |
| **Bottleneck — в последовательных вызовах** | 5 вызовов Author, а не 5 проверок дубликатов. Split не решает эту проблему. |
| **Author уже простой** | 61 строка system prompt. Разделение на 2 агента по ~40 строк — рост сложности без выигрыша. |
| **Два агента = двойная поддержка** | Изменения в шаблонах, frontmatter, Mermaid-правилах нужно синхронизировать в 2 местах. |
| **Размытие зон ответственности** | Researcher создаёт synthesis, основанный на существующих concepts — это create или update? |
| **Editor idle 95% времени** | Страницы создаются и рецензируются один раз. Система нацелена на создание, не на постоянное обновление. |

### Вывод по Author Split

**❌ Не внедрять.** Решает проблему, которой нет в проекте (сложная логика update). Добавляет сложность без устранения реального bottleneck (последовательные вызовы).

**Альтернатива**: вместо Drafter+Editor — оставить одного Author, но добавить флаг `--mode update` для случаев, когда нужно обновить существующую страницу. Это 5 строк в существующем agent-файле, а не 2 новых агента.

## 5. Дизайн Author v2 (Batch-capable)

### Принципы

1. **Один agent-файл** (`author.md`), два режима работы
2. **Batch mode** — для новых спецификаций (основной use case)
3. **Single mode** — для отдельных страниц, обновлений, Researcher
4. **Context-aware**: если PDF слишком большой (>150K знаков) → автоматический sequential fallback
5. **Совместимость**: все 5 call sites продолжают работать без изменений

### Структура нового author.md

```
# Agent: Author v2 ObsidianDB

## Режимы работы

### MODE A: Batch (пакетная обработка спецификации)
Триггер: "Author: пакет <путь-к-PDF>"
Процесс:
  1. Прочитай PDF ОДИН раз
  2. Составь план: какие страницы нужны (summary + 3-8 concepts + 0-1 entity)
  3. Создай summary (содержит ссылки на будущие concepts)
  4. Создай concepts (ссылаются на summary)
  5. Создай entity если нужно
  6. Обнови все индексы одним проходом
  7. /lint

### MODE B: Single (одна страница)
Триггер: "Author: создай concept <имя>" (совместимость с текущим)
Процесс: как сейчас, без изменений
```

### Ограничение размера (context window guard)

```
Если извлечённый текст PDF > 150 000 знаков:
  → Sequential mode: создать summary → concepts по одному,
    но передавать summary как контекст для каждого concept
    (context reuse без риска переполнения)
Иначе:
  → Batch mode: всё за один проход
```
