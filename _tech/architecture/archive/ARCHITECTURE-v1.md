## 1. Общая архитектура

### 1.1 Философия проекта

ObsidianDB — это AI-управляемая база знаний по телекоммуникационным спецификациям, работающая внутри Obsidian-хранилища. Главный принцип: **AI-агенты как активные участники knowledge-менеджмента**, а не пассивные ассистенты.

Ключевая инновация — **Reviewer v2**, который сверяет утверждения на wiki-страницах с эталонными текстами спецификаций (извлечёнными через SpecExtractor), а не только с другими wiki-страницами. Это создаёт **эталонную цепочку**: `PDF → TXT → wiki → Review`.

### 1.2 Принципы проектирования

| Принцип | Реализация |
|---|---|
| **Разделение прав** | Чёткое разграничение: read-only (`Specifications/`, `Clippings/`, `.obsidian/`), writable (`wiki/`, `outputs/`, `notes/`) |
| **Эталонная цепочка** | `Specifications/` (PDF) → `specs-extracted/` (TXT) → wiki. Wiki не может противоречить спецификациям |
| **Модульные агенты** | 8 узкоспециализированных агентов, каждый с одной зоной ответственности |
| **Пайплайн-скиллы** | Skills оркестрируют нескольких агентов для сложных процессов |
| **Всё под lint** | Каждое изменение завершается `/lint` — 0 битых ссылок, 0 сирот |
| **Provenance** | Каждое утверждение маркируется: `^[extracted]`, `^[inferred]`, `^[ambiguous]`, `^[todo]` |
| **Шаблонизация** | 6 Obsidian-темплейтов для единообразия всех типов страниц |

### 3.2 Спецификация каждого агента

#### Author
| Параметр | Значение |
|---|---|
| **Файл** | `.claude/agents/author.md` |
| **Роль** | Создание новых wiki-страниц |
| **Вход** | Исходный материал (PDF, clipping, заметка) |
| **Выход** | `.md` страница в `wiki/` с frontmatter, wikilinks, Mermaid |
| **Триггер** | `/ingest`, запрос на создание страницы |
| **Ключевые правила** | Шаблоны `.obsidian/templates/`, префикс `wiki/`, provenance-пометки |

**Алгоритм работы:**
1. Прочитать исходный материал
2. Определить тип (concept/entity/summary/synthesis)
3. Использовать шаблон `t-*.md`
4. Написать: определение → детали → связи → источники
5. Добавить Mermaid (если уместно)
6. Расставить wikilinks с префиксом `wiki/`
7. Обновить индексы (`wiki/index.md`, раздельный index)
8. Выполнить `/lint`
9. Обновить `Roadmap.md`

#### Reviewer v2
| Параметр | Значение |
|---|---|
| **Файл** | `.claude/agents/reviewer.md` |
| **Роль** | 3-проходная проверка качества |
| **Вход** | Wiki-страница + эталонные спецификации |
| **Выход** | Отчёт: CRITICAL/HIGH/MEDIUM/LOW ошибки + вердикт |
| **Триггер** | `/review`, после Author |
| **Инновация** | Сверка с `specs-extracted/` а не с другими wiki |

**Эталонная цепочка:**
```
Specifications/PDF  ──SpecExtractor──►  specs-extracted/TXT  ──Reviewer──►  wiki/.md
       (истина)                        (читаемый эталон)                (проверено)
```

**3 прохода:**
1. **Pass 1 — Техническая точность**: сверка FID/CLA/INS/AID/SW/размеров EF со спецификациями
2. **Pass 2 — Структура**: frontmatter, заголовки, Mermaid, callouts
3. **Pass 3 — Связность**: ≥3 входящих, ≥3 исходящих ссылок

**Критичность ошибок:**
| Уровень | Критерий | Пример |
|---|---|---|
| CRITICAL | Неверный FID/CLA/INS/AID/SW | `EF_IMSI = 6F05` вместо `6F07` |
| HIGH | Неверный размер EF, структура данных | `EF_ICCID = 20 байт` вместо `10` |
| MEDIUM | Неверный TERMINAL PROFILE бит | Бит 2 вместо бита 3 |
| LOW | Опечатка, стиль | — |
| NEEDS_SPEC | Невозможно проверить без спецификации | — |

**Найденные критические ошибки (Фаза 4):**
- DF_5GS FID: `6Fxx` → `4Fxx` (28 мест) — TS 31.102 §4.4.11
- CSIM AID: `A0..87.10.06` → `A0..03.43.10.02` — TS 101 220 Annex M
- TERMINAL PROFILE INS: `70` → `10` — TS 102 221
- +6 других исправлений

#### Linker
| Параметр | Значение |
|---|---|
| **Файл** | `.claude/agents/linker.md` |
| **Роль** | Анализ и улучшение связности графа знаний |
| **Вход** | `wiki/` (все страницы) |
| **Выход** | Предложения wikilinks, мосты между кластерами |
| **Триггер** | `/lint` (обнаружение сирот), `/ingest` шаг 6 |

**Алгоритм:**
1. Найти orphan-страницы (0 входящих)
2. Найти страницы с <3 входящих ссылок
3. Предложить 2-3 wikilinks с других страниц
4. Найти изолированные кластеры
5. Предложить мосты между кластерами
6. `/lint` для верификации

#### Librarian
| Параметр | Значение |
|---|---|
| **Файл** | `.claude/agents/librarian.md` |
| **Роль** | Каталогизация новых материалов |
| **Вход** | Новый файл в `Specifications/` или `!INCOMING/` |
| **Выход** | Summary + Concepts + Entities + обновлённый Roadmap |
| **Триггер** | `/ingest`, появление файлов в `!INCOMING/` |

#### Researcher
| Параметр | Значение |
|---|---|
| **Файл** | `.claude/agents/researcher.md` |
| **Роль** | Глубокие исследования (15-50+ KB) |
| **Вход** | Тема + все релевантные источники |
| **Выход** | `wiki/research/<slug>.md` |
| **Триггер** | Запрос на глубокое исследование |

**Отличия от других форматов:**
| | Summary | Synthesis | Research |
|---|---|---|---|
| Глубина | Конспект | Кросс-анализ | Исчерпывающий разбор |
| Объём | 1-3 KB | 5-15 KB | 15-50+ KB |
| Источников | 1-2 | 3-7 | 5-20+ |

#### Formatter
| Параметр | Значение |
|---|---|
| **Файл** | `.claude/agents/formatter.md` |
| **Роль** | Конвертация Markdown → HTML |
| **Вход** | `.md` файл из `wiki/` |
| **Выход** | `outputs/<имя>.html` с тёмной темой, Mermaid, callouts |
| **Триггер** | `/format-html` |
| **Критично** | Mermaid-правила (literal `>` не `&gt;`, только ASCII, без `<br/>`) |

#### SpecExtractor
| Параметр | Значение |
|---|---|
| **Файл** | `.claude/agents/specextractor.md` |
| **Роль** | Извлечение PDF → TXT для Reviewer |
| **Вход** | PDF из `Specifications/` |
| **Выход** | `.txt` в `specs-extracted/` + `INDEX.md` + `.meta.json` |
| **Триггер** | Первый запуск, новый PDF через `!INCOMING`, апдейт спецификации |

**Алгоритм:**
1. Сканировать `Specifications/` рекурсивно
2. Для каждого PDF без TXT-копии: `PyPDF2.PdfReader` → извлечь все страницы
3. Сохранить с сохранением структуры директорий
4. Создать/обновить `INDEX.md` с картой содержимого
5. Записать метрики в `.meta.json`
6. Обновить Roadmap

> **Будущее**: замена PyPDF2 на Docling (через 3gpp-crawler) — см. `_tech/3gpp-crawler-integration-plan.md`. Docling сохраняет таблицы, заголовки и provenance-координаты (секция/таблица/строка) против плоского TXT у PyPDF2.

#### SpecDownloader (NEW — Фаза 5)
| Параметр | Значение |
|---|---|
| **Файл** | `.claude/agents/specdownloader.md` |
| **Роль** | Скачивание спецификаций 3GPP/ETSI напрямую в `!INCOMING/` |
| **Вход** | Номер спецификации (TS/TR) + релиз |
| **Выход** | PDF в `Specifications/!INCOMING/` + опционально структурированный .md/.json |
| **Триггер** | `SpecDownloader: скачай TS 31.102`, обновление Roadmap-спецификаций |
| **Инструмент** | `spec-crawler` CLI (глобально через `uv tool install`) |

**Поддерживаемые источники:**
| ✅ 3GPP TS/TR | 31.xxx, 22.xxx, 23.xxx, 24.xxx, 33.xxx, 35.xxx, 38.xxx |
| ✅ ETSI TS | 102.221, 102.223, 102.225, 102.226 (через 3GPP FTP) |
| ❌ GSMA | SGP.22, SGP.32 — не на 3GPP FTP |
| ❌ ISO | 7816 — не на 3GPP FTP |
| ❌ GlobalPlatform | Не на 3GPP FTP |

**Как это меняет INCOMING-пайплайн:**
```
Раньше:  пользователь ищет PDF вручную → скачивает → кладёт в !INCOMING/
Теперь:  SpecDownloader скачивает автоматически → PDF в !INCOMING/
         → Librarian сортирует → /ingest → /lint → Roadmap
```

### 4.2 Спецификация каждого skill

#### /lint (lint-wiki)
| Параметр | Значение |
|---|---|
| **Файл** | `.claude/skills/lint-wiki/SKILL.md` |
| **Частота** | После КАЖДОГО изменения в wiki/ |
| **Инструменты** | Glob, Grep, Read |

**5 проверок:**
1. Битые ссылки: `[[wiki/...]]` → существует ли целевой файл
2. Сироты: страницы без входящих wikilinks (кроме index.md)
3. Frontmatter: tags, type, status, created, updated, sources
4. Противоречия: конфликтующие FID/CLA между страницами
5. Пробелы: темы без покрытия

#### /ingest (ingest)
| Параметр | Значение |
|---|---|
| **Файл** | `.claude/skills/ingest/SKILL.md` |
| **Sub-agents** | Author (шаги 2-5), Linker (шаг 6) |

**8 шагов:**
1. Прочитать исходный файл (PDF → PyPDF2)
2. Создать summary в `wiki/summaries/`
3. Извлечь концепты → `wiki/concepts/`
4. Зафиксировать сущности → `wiki/entities/`
5. Создать synthesis (если кросс-тематика)
6. Расставить ссылки (Linker)
7. Обновить Roadmap.md
8. Выполнить `/lint`

#### /review (review)
| Параметр | Значение |
|---|---|
| **Файл** | `.claude/skills/review/SKILL.md` |
| **Sub-agents** | Reviewer (Pass 1), Reviewer (Pass 2), Linker (Pass 3) |

**Критерии reviewed:**
- Нет противоречий со спецификациями
- Все `^[extracted]` ссылаются на конкретный раздел
- Все FID/CLA/INS/TAG проверены
- Структура: Определение → Детали → Связи → Источники
- Минимум 3 входящих и 3 исходящих ссылки
- Mermaid-диаграмма где уместно

#### /format-html (format-html)
| Параметр | Значение |
|---|---|
| **Файл** | `.claude/skills/format-html/SKILL.md` |
| **Sub-agents** | Formatter |
| **Критично** | Mermaid-правила — ЕДИНСТВЕННЫЙ источник истины |

**Mermaid-правила (на все агенты):**
- Только literal ASCII стрелки: `->>` не `&gt;`
- Без box-drawing chars: `═══` `───` → `===` `---`
- Без emoji: `✅❌🔑` → `OK/FAIL/KEY`
- `/` вместо `||`
- Раздельные `Note` вместо `<br/>`

|---|---|

## 9. Сильные стороны архитектуры

1. **Эталонная цепочка** — уникальная фича: PDF → TXT → wiki, предотвращает копирование ошибок
2. **Reviewer v2** — сверка со спецификациями а не с другими wiki, нашёл 9 критических ошибок
3. **Модульность агентов** — каждый агент имеет одну зону ответственности, легко заменять/улучшать
4. **Skills как оркестраторы** — отделяют «что делать» от «как делать»
5. **Provenance-маркировка** — каждое утверждение имеет traced происхождение
6. **100% review coverage** — все 122 контентные страницы проверены
7. **0 битых ссылок** — автоматический lint после каждого изменения
8. **Разделение прав** — read-only на исходники, защита от случайных изменений
9. **SpecDownloader** — автоматическое скачивание 3GPP спецификаций, конец ручному поиску

## 10. Зоны для улучшения

1. **Отсутствие version control** — нет git (отложено)
2. **PyPDF2 теряет таблицы** — замена на Docling запланирована (см. `3gpp-crawler-integration-plan.md`)
3. **Нет CI/CD** — lint/review только по требованию
4. **Монолитный CLAUDE.md** — 240+ строк, мог бы быть модульным
5. **Нет валидации frontmatter** — только визуальная проверка
6. **Нет метрик качества** — нет чисел по coverage, complexity, link density
7. **Отсутствие бэкапов** — нет стратегии резервного копирования
8. **.claude/CLAUDE.md устарел** — говорит «6 sub-agents, 86 страниц» (на самом деле 8 и 129)
9. **outputs/STATUS_AND_PLAN.md устарел** — не отражает новые страницы
